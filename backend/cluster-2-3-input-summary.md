## 📋 输入数据结构

### 1. 聚类基本信息
```json
{
  "cluster_id": "cluster_2_3",
  "cluster_name": "数据使用、加工与展示"
}
```

### 2. 成熟度级别定义（5个级别）

每个级别包含3个字段：
- **name**: 级别名称（初始级、可重复级、已定义级、可管理级、优化级）
- **description**: 详细描述（包含原始条目要求、现状、管理特征）
- **key_practices**: 关键实践列表（数组）

#### Level 1 - 初始级（3条关键实践）
- 未建立企业级数据服务管理体系和专职团队
- 高敏感性数据项随意存储在终端和移动介质
- 展示高敏感性数据项时未实施脱敏处理

#### Level 2 - 可重复级（3条关键实践）
- 展示高敏感性数据项时原则上实施脱敏
- 加工高敏感性数据项时明确保护措施并进行审批
- 从生产环境提取敏感级以上数据时实施严格审批

#### Level 3 - 已定义级（4条关键实践）
- 建立企业级数据服务管理体系、制定规范并建立专职团队
- 处理敏感级以上数据或开展重大活动前开展数据安全评估
- 开展敏感数据清洗转换、汇聚融合时采用匿名化、去标识化措施
- 加工产生新数据项时，经评估后动态调整敏感性标识

#### Level 4 - 可管理级（3条关键实践）
- 对敏感级以上数据严格实施授权管理，制定访问闭环机制并审计
- 根据业务需要明确数据保存期限，统一规范管理高敏数据在终端的存储
- 对确需不脱敏展示或导出的高敏感性数据项实施统一规范管理和监控

#### Level 5 - 优化级（3条关键实践）
- 自动监测数据汇聚融合导致的级别变化并及时自动调整保护措施
- 加工产生新数据项时，系统自动评估并调整敏感性标识
- 基于数据分析持续优化数据加工分析流程和服务管理体系

## 🎯 问卷生成Prompt结构

### 核心指令

1. **角色定位**
   - "你是一名资深IT咨询师，专注于调研问卷设计"

2. **固定输出要求**
   - 固定生成5个问题
   - 每个问题从不同维度提问

3. **5个维度（必须严格遵守）**
   - 维度1【政策与制度层面】：是否有正式的政策/流程文档
   - 维度2【执行与实施层面】：实际操作中是否落实相关要求
   - 维度3【监控与度量层面】：是否有监控机制和量化指标
   - 维度4【持续改进层面】：是否有定期评审和优化机制
   - 维度5【证据与合规层面】：是否有留存记录和审计证据

4. **交叉验证机制**
   - "通过不同维度的问题，交叉验证用户答案的一致性，识别潜在的虚假陈述"

5. **题型限制**
   - 只使用SINGLE_CHOICE（单选题）和MULTIPLE_CHOICE（多选题）
   - **禁止使用RATING评分题**

6. **输出格式**
   - 严格输出JSON格式
   - 不要添加任何注释或markdown标记
   - 单选题必须有5个选项对应5个成熟度级别

## 💡 关键发现

### 1. 数据传递方式
- **矩阵任务**的`result.content`字段存储完整的矩阵数据（16行聚类）
- **问卷任务**的`input`字段只有`matrixTaskId`，需要从数据库查询矩阵结果
- 避免了在任务input中存储大量数据（数据库优化）

### 2. Prompt填充策略
- **分批处理**：每次只传入1个聚类，生成5个问题
- **结构化输入**：
  - 聚类ID和名称（用于识别）
  - 5个成熟度级别（用于生成单选题的5个选项）
  - 每个级别的关键实践（用于生成多选题选项）

### 3. 交叉验证设计
- 5个维度的问题可以相互验证
- 例如：如果政策层面选Level 4，但执行层面什么都没做 → 矛盾
- 识别虚假或不一致的答案

### 4. 输出约束
- **固定格式**：必须输出JSON，不允许markdown或注释
- **固定数量**：必须5题，不能多也不能少
- **固定题型**：只用单选和多选，禁止评分题
- **固定字段**：每个问题的dimension必须明确标注维度

## 🎓 总结

**问卷生成器的输入** = **1个聚类的完整成熟度矩阵**（5个级别 × 每个级别3-4条关键实践）

**AI的任务** = 基于这5个级别的信息，生成5个不同维度的问题（用于交叉验证）

**核心特点**：
1. 结构化输入（清晰的5级成熟度定义）
2. 明确约束（固定5题、固定题型、固定格式）
3. 交叉验证设计（5个维度相互印证）
4. 容错机制（三模型并行、降级填充）
