# Story 2.9: Report Viewing and Export

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a ThinkTank user,
I want to view and export the report generated by a workflow,
so that I can share the outcome outside the conversation.

## Acceptance Criteria

1. Given a workflow has generated at least one report section, when the user opens the document drawer or report view, then the report renders as structured Markdown with title, sections, AI label, and generation metadata, and the user can export the current report state without waiting for workflow completion.
2. Given the user chooses Markdown export, when the export completes, then a Markdown file is produced with the same visible structure as the report view, a success toast confirms the export without interrupting the workflow, and audit event `thinktank.output.exported` is emitted with output id, format `markdown`, tenant, actor, and AI-label metadata presence.
3. Given the user chooses PDF export, when the export completes, then a PDF file is produced using CJK-capable fonts and preserving AI labeling, export failures show a recoverable error with next-step guidance, and successful export emits audit event `thinktank.output.exported` with output id, format `pdf`, tenant, actor, and AI-label metadata presence.

### Previous Story Intelligence

- Story 2.8 introduced `workflow_outputs`, `AdvisoryWorkflowOutput`, `AdvisoryWorkflowOutputRepository`, guarded session output routes, Next proxy routes, typed frontend output helpers, and `AdvisoryDocumentDrawer`.
- Story 2.8 review fixes are non-negotiable regression guards: one active draft per tenant/session, transactional append/complete, active-session gating for mutations, persisted assistant source validation, empty draft rejection, audit failure propagation, explicit-continue append behavior, final-step completion, drawer listener cleanup, event-level decision option preservation, and avoiding unnecessary session-message refetches.
- Export must read the persisted `workflow_outputs` snapshot. Do not export directly from React state because that bypasses tenant scoping, backend AI metadata validation, and `thinktank.output.exported` audit.
- `getSessionOutput` creates an empty draft when none exists. Export must not call that path blindly if it would create and then export an empty report.
- `completeOutput` requires an active session. Export must support both draft and completed outputs and therefore needs its own read path rather than reusing completion gating.

### Backend Implementation Guidance

- Prefer adding a dedicated `AdvisoryOutputExportService` in `backend/src/modules/advisory/outputs/` and registering it in `AdvisoryModule`.
- Export lookup should validate module access, tenant ownership, and session ownership, then select the active draft if present or the latest completed output for that tenant/session.
- Required exported output predicates:
  - `sections.length > 0`
  - each exported section preserves visible `[AI Generated]`
  - `aiLabelMetadata.visible_label` or equivalent is `[AI Generated]`
  - metadata confirms `ai_generated === true` and `machine_readable === true`
- Suggested result contract:
  - `buffer: Buffer`
  - `contentType: 'text/markdown; charset=utf-8' | 'application/pdf'`
  - `fileName: string`
  - `outputId`, `format`, `sectionCount`
- Suggested event metadata:
  - `format`, `workflow_key`, `section_count`, `ai_label_metadata_present: true`
  - no `content_markdown`, `sections`, `raw_content`, `prompt`, `messages`, or provider raw payloads
- Use `Content-Disposition: attachment; filename="..."` plus sanitized ASCII fallback filenames.

### Frontend Implementation Guidance

- Add export buttons inside the drawer action area, likely before the close button. Keep compact icon+text or icon-only with accessible labels.
- Export state should be explicit: `exportingFormat`, `exportError`, and a single handler in `AdvisoryWorkspaceShell` that calls the typed output client.
- A successful export should show a toast and leave current workflow state untouched: no drawer close, no message reload, no decision-option reset, no current-step mutation.
- Recoverable errors should be visible in the drawer until dismissed or a later successful export, with `role="alert"` and practical next-step guidance.
- Tests must use role, label, text, title, status, and alert selectors. Do not add production `data-testid`.
