# Product Requirements Document - Csaas

**Author:** 27937
**Date:** 2025-12-23

## Executive Summary

**Csaas（AI驱动的IT咨询成熟度评估SaaS平台）** 旨在变革IT咨询行业的成熟度评估流程。通过三模型AI协同架构和人机协作机制，平台将传统需要数周完成、高度依赖个人经验的咨询项目，转化为标准化、可规模化、质量可控的智能服务。

**解决的核心问题：**

传统IT咨询成熟度评估面临三大痛点（按严重程度排序）：

1. **质量不稳定（最核心）** - 评估结果高度依赖咨询师个人经验和状态，同一标准不同咨询师理解差异可达30%以上，导致客户信任度低、复购率低
2. **效率瓶颈** - 主咨询师需手工分析IT标准文档（200-300页）、聚类要求、设计问卷（50-100题）、解读结果，单个项目耗时2-3周，其中60%是重复性工作
3. **难以规模化** - 人工流程限制咨询公司同时服务客户数量（通常≤3个并行项目/每个主咨询师），严重制约业务增长

**市场现状与竞争定位：**

当前市场上的替代方案主要有三类：
- **纯人工咨询**（传统方式）- 质量高但成本高、周期长、难规模化
- **问卷工具类SaaS**（如问卷星、Typeform）- 只解决问卷收集，不解决标准分析和专业性
- **大型咨询公司内部工具**（如McKinsey Lilli、BCG GENE）- 仅供内部使用，中小咨询公司无法获得

**Csaas填补的市场空白：** 为中小型IT咨询公司提供"AI专家助理"级别的智能化支持，既保留人工咨询的专业性和定制化，又获得接近自动化工具的效率和一致性。

**市场规模估算：**

- **TAM（总可达市场）：** 全国中小型IT咨询公司约500-800家，年均成熟度评估项目15个/公司，平均项目收费30万元 → 总市场规模约22.5-36亿元
- **SAM（可服务市场）：** 按平台收费为项目费用的10-15%估算 → 可服务市场规模约2.25-5.4亿元
- **数据飞轮临界点：** 预计积累100个项目后，对标数据开始有显著价值；500个项目后形成行业benchmark；1000+项目后形成强壁垒（竞争对手需2-3年追赶）

**目标用户：**

- **主要用户：** 中小型IT咨询公司的主咨询师（3-20人规模的团队）
- **次要用户：** 企业项目经理（接收问卷、查看报告）
- **最终受益者：** 被评估企业的各角色人员（填写问卷）

**核心价值主张：**

**对咨询公司的价值：**
- **降本50%** - AI自动化处理标准分析、聚类、问卷生成、评分等重复性工作（假设：2-3周人工工作 → 1周AI生成+人工审核，人力成本减半）
- **提质保证** - 三模型交叉验证（80%相似度门槛）确保输出质量，理论上优于单一咨询师（单点偏差被多数票纠正）
- **规模化赋能** - 咨询师从"手工全做"转变为"AI生成+专家审核"，并行项目能力提升3-5倍（假设：审核时间为生成时间的20-30%）
- **持续增值** - 订阅制获得IT标准库持续更新、行业对标数据访问、AI模型优化

**对最终企业客户的价值：**
- **问卷设计更科学** - 三模型交叉验证确保100%覆盖标准要求，无遗漏、无冗余
- **评估结果更客观** - 基于IT标准规范而非咨询师主观经验，一致性和可复现性更高
- **行业对标更精准** - 基于去标识化真实项目数据的百分位排名，而非理论估算
- **交付周期更短** - 2-3周 → 1周（问卷设计+数据收集+分析报告）
- **建议更可落地** - AI基于标准要求自动生成成熟度跃迁的具体落地措施清单

### 产品独特性

**1. 三模型众包+互审架构（技术护城河）**

**核心原理：独立验证与最小可验证集合**

传统单一AI模型面临的根本问题：
- **不确定性无法自证** - 模型输出是概率分布，说"我85%确定"的置信度本身可能是错的（高置信度的错误答案）
- **偏差模式固定** - 每个模型基于特定训练数据和架构，有固定的"错误倾向"（如GPT对某类问题理解偏差，Claude对另一类问题理解偏差）

**Csaas的解决方案基于两个科学原理：**

**原理1：独立验证原理**（源自科学方法论）
- 单一观察者无法验证自身的正确性，需要多个独立观察者达成共识来建立客观性
- 类比：科学研究中的同行评审（peer review）—— 三个独立审稿人达成一致，论文更可能正确

**原理2：最小可验证集合**（源自投票理论和拜占庭将军问题）
- 一个模型：无法自我验证
- 两个模型：意见分歧时无法裁决（50% vs 50%）
- **三个模型**：可通过多数票裁决（2:1或3:0），是最小的可验证集合
- 四个或更多：边际收益递减，成本线性增加

**架构设计：**

```
三模型并行生成（GPT-4、Claude、国产模型）
  ↓
分层相似度验证（而非单一阈值）：
  ├─ Level 1: 结构一致性 ≥ 90%（核心，如聚类的类别数量和划分逻辑）
  ├─ Level 2: 语义等价性 ≥ 80%（重要，使用embedding计算语义相似度）
  └─ Level 3: 细节一致性 ≥ 60%（次要，措辞和排序的差异可接受）
  ↓
质量判定：
  ├─ 全部通过 → 投票选最佳结果（高置信度）
  └─ 任一层级不通过 → 半自动诊断性重试：
      ├─ AI分析差异点（如"模型A将条款X归为类别1，模型B归为类别2，差异原因可能是..."）
      ├─ 代码结构化呈现并提示主咨询师
      ├─ 主咨询师选择策略（补充材料/调整提示/换模型/接受并降级）
      └─ 仍不通过 → 降级为"初稿"（高亮提示需要额外关注）
```

**创新价值：**
- 将AI"置信度"从主观自评（"我85%确定"）转化为客观的**独立验证共识**（"三个独立专家一致性达90%"）
- 三模型不是"越多越好"的堆砌，而是基于验证理论的**最优配置**
- 分层相似度验证确保"抓大放小"——核心逻辑必须一致，表述细节可以灵活

**人工审核的工作量说明（基于验证而非重做）：**

AI生成后，主咨询师的审核工作不是"重新做一遍"，而是"验证AI的逻辑"（类比代码审查中的逻辑验证）：
- **高置信度结果（≥85%）**：AI提供聚类逻辑说明+原文索引，咨询师快速扫描验证（预计5-10分钟/环节）
- **中等置信度结果（75-85%）**：AI高亮差异点+分层相似度详情，咨询师重点审核争议条目（预计15-20分钟/环节）
- **低置信度结果（<75%）**：AI展示三模型分歧详情+诊断分析+原文对照，咨询师深度验证或手工调整（预计30-60分钟/环节）

**总审核时间估算：** 每个项目5个AI生成环节（综述、聚类、矩阵、问卷、落地措施），假设平均置信度80%，总审核时间约2-3小时，相比传统手工2-3周工作量，效率提升90%+。

**AI与代码的职责边界（基于能力本质匹配原则）：**

**划分原则：** 不是"哪个更方便"，而是"哪个本质上更适合"

- **AI负责"理解和生成"** - 自然语言理解、语义识别、内容生成是AI的本质能力（基于概率分布和模式识别）
- **代码负责"验证和编排"** - 精确计算、条件判断、流程控制是代码的本质能力（基于确定性逻辑和规则执行）
- **协作处理"质量保证"** - AI生成多样性（3个独立结果）+ 代码验证一致性（相似度计算）；AI解释推理过程（逻辑说明）+ 代码编排审核流程（渐进式披露）

| 层级 | AI职责（模式识别与生成） | 代码职责（规则执行与管理） | 协作机制 |
|------|----------------------|--------------------------|---------|
| **内容生成** | 理解标准文档、聚类条款、生成问卷、打分评估、生成落地措施 | 无 | AI独立完成 |
| **质量验证** | 独立生成3份结果、计算语义embedding | 计算条目级相似度（集合重合度）、计算语义距离（余弦相似度）、投票选最佳、触发重试/降级、覆盖率检查（300条→280条，列出缺失20条） | AI生成+代码验证 |
| **诊断分析** | 分析三模型差异点、推测可能原因（需要语义推理） | 将诊断结果结构化呈现、触发主咨询师决策流程 | AI分析+代码编排 |
| **人工审核** | 生成审核辅助信息（逻辑说明、原文对照、挑战性问题） | 渐进式披露流程编排、高风险项识别算法、审核质量追踪、预警触发 | AI辅助+代码编排 |
| **数据管理** | 无 | 版本管理、向下兼容、去标识化、对标库构建、权限控制 | 代码独立完成 |

**人工审核的必要性（基于三个原理）：**

1. **AI不确定性原理** - 大语言模型本质是概率分布，即使三模型一致也可能集体误判（如标准文档本身有歧义导致三个模型都理解错误）
2. **可问责性要求** - 咨询报告需要咨询师签字负责，法律和职业伦理要求人类专家对输出承担责任，纯AI输出无法问责
3. **领域知识互补** - AI理解"通用语义"，咨询师理解"行业特殊性"（如某银行的特殊合规要求），两者结合才能产出高质量方案

**2. 渐进式披露的五道审核关卡（防误导设计）**

系统设计了三阶段审核流程防止咨询师被AI误导：
- **独立思考阶段** - 先展示原始标准文档，激活咨询师独立判断（防止锚定效应）
- **对比发现阶段** - 展示AI结果+一致度+逻辑说明，高亮高风险项（置信度<75%、三模型分歧>40%、涉及合规条款）
- **深度验证阶段** - 针对高风险项，展示AI聚类逻辑说明+原文对照+挑战性问题引导（如"为什么条款A和条款B被聚在一起？它们的共同点是什么？"）

配合四个防止流于形式的机制：
1. **审核质量追踪** - 审核时间<10分钟/环节 + 零修改率 + 高风险项全接受 → 预警提示
2. **能力校准测试** - 每月混入1-2个AI故意错误，咨询师能发现 → 能力合格 ✅；未发现 → 触发培训提醒 ⚠️
3. **同行互审** - 重要项目由两个主咨询师交叉审核，意见分歧时需仲裁
4. **创始人终极审核** - Demo阶段100% → 成长期30% → 成熟期10%（建立质量baseline）

**3. 数据主权与隐私设计（信任基础）**

**数据分级策略：**

| 数据类型 | 归属权 | 使用权限 | 示例 |
|---------|--------|----------|------|
| **独占数据** | 咨询公司100%拥有 | 平台无法访问、无法用于对标 | 企业名称、具体问卷回复、项目报告 |
| **共享数据** | 客户明确授权后共享 | 去标识化后用于行业对标 | 成熟度评分、差距分析（不含企业识别信息） |
| **平台资产** | 平台拥有 | 订阅期内咨询公司可使用 | IT标准库、咨询领域模板、对标算法 |

**隐私保护机制：**
- **Opt-in授权** - 默认不参与对标数据库，需客户明确勾选"允许匿名数据用于行业对标"
- **分层订阅** - 提供"完全私有版"订阅选项（价格+20%），数据100%不共享
- **去标识化标准** - 移除企业名称、地址、联系方式等18类识别信息，符合GDPR/PIPL要求
- **数据审计** - 咨询公司可随时查看"哪些数据被用于对标"的审计日志

**4. 订阅制知识服务+数据飞轮（商业模式护城河）**

平台不是一次性工具，而是持续订阅服务：
- **知识资产** - IT标准库持续更新（ISO/ITIL/COBIT等每年更新）、咨询领域模板优化、行业benchmark积累
- **数据网络效应** - 去标识化对标数据随项目数量指数级增值：
  - 100个项目 → 基础对标（行业平均水平）
  - 500个项目 → 细分行业对标（如"金融vs制造"）
  - 1000个项目 → 精准百分位排名（如"你在同行业top 15%"）
- **锁定机制** - 不续订即失去更新和对标数据访问权，离开成本递增（使用1年后离开成本 ≈ 重新积累1年数据的机会成本）

**竞争壁垒的时间维度：**
- **6个月（Demo期）**：完成前50个项目，验证产品-市场匹配
- **12-18个月（成长期）**：积累300-500个项目，对标数据开始显现价值，形成初步壁垒
- **24-36个月（成熟期）**：积累1000+项目，行业benchmark成为标准，竞争对手追赶需2-3年
- **先发优势窗口**：预计18-24个月内建立数据壁垒，之后新进入者难以获得足够数据样本

**5. 赋能而非替代的定位（组织变革策略）**

**产品设计中的赋能机制：**
- **个人品牌建设** - 专家主页显示完成项目数、客户评分、擅长领域、平台认证徽章
- **知识产权保护** - 咨询师优化的内容标记为"专家版本"，他人使用需署名/分成（如优化后的问卷模板）
- **透明度控制权** - 咨询师决定客户看到的报告形式：
  - 完全黑盒："专家咨询报告"（客户看不到AI痕迹）
  - 半透明："AI辅助的专家分析"（标注AI生成部分）
  - 完全透明："AI生成+专家审核"（完整展示流程）

**渐进式导入策略（降低组织抵抗）：**
1. **第一阶段（1-3个月）**：只用AI做标准检索和综述（最枯燥的工作，咨询师最愿意放手）
2. **第二阶段（4-6个月）**：逐步扩展到聚类、问卷生成（咨询师已建立信任）
3. **第三阶段（7-12个月）**：启用全流程AI辅助+人工审核（咨询师习惯协作模式）

**营销话术调整：**
- ❌ "AI替代人工提高效率"
- ✅ "AI让咨询师能力提升10倍，从单兵作战到超级个体"
- ✅ "把咨询师从重复劳动中解放，聚焦高价值的客户沟通和方案定制"

## 核心技术架构决策

为确保三模型架构的工程可行性和质量保障的可验证性，Csaas在以下关键技术领域做出了明确的架构决策：

### 三模型并行调用架构

**架构模式：异步队列 + 多级降级**

```
用户提交任务 → Task Manager创建任务记录
  ↓
三个Worker并行调用API（独立执行，互不等待）
  ├─ Worker 1: GPT-4 → 结果写入数据库
  ├─ Worker 2: Claude → 结果写入数据库
  └─ Worker 3: 国产模型 → 结果写入数据库
  ↓
Aggregator监听：三个结果都完成时触发
  ↓
相似度计算 → 投票选最佳 → 通知用户
```

**关键设计决策：**

- **异步执行模式** - 咨询师提交任务后可继续其他工作，完成后通知（邮件/短信/站内消息）
- **超时策略** - 单Worker超时10分钟，整体任务超时30分钟
- **多级降级** - 30分钟后根据完成情况自动降级（3个成功→2个成功→1个成功→全失败手工模式）
- **API密钥管理** - 每个模型配置双密钥（primary + fallback），限流时自动切换
- **容错机制** - 单个Worker失败不影响其他，失败时自动重试（最多3次，间隔5-10分钟）

### 相似度计算算法

**分层验证的技术实现：**

**Level 1：结构一致性计算（≥90%）**
- **聚类场景** - 使用匈牙利算法（Hungarian Algorithm）找最优类别匹配，计算Jaccard相似度
  - 示例：模型A识别5个类别，模型B识别5个类别 → 一对一匹配 → 计算每对类别的条目重合度
  - 集合比较忽略顺序（用set运算）

- **矩阵场景** - 结构化单元格对比，计算维度一致性和单元格填充完整性

**Level 2：语义等价性计算（≥80%）**
- 使用统一embedding模型（OpenAI text-embedding-ada-002）生成语义向量
- 计算两两余弦相似度：sim(1,2), sim(1,3), sim(2,3)
- 平均相似度 = (sim(1,2) + sim(1,3) + sim(2,3)) / 3
- 避免"三个模型各自embedding空间不同"的比较误差

**Level 3：细节一致性计算（≥60%）**
- 字符级Levenshtein距离（编辑距离）
- 用于表述细节对比，阈值较宽松

**投票选最佳算法：**
- 当三个结果通过相似度验证后，选择"平均语义质量分最高"的结果
- 质量分 = 专业术语覆盖度 × 0.4 + 逻辑连贯性 × 0.3 + 原文引用准确性 × 0.3

### 版本管理与向下兼容

**数据模型设计：**

```
咨询领域版本表（consulting_domains）
  ├─ domain_id（领域唯一标识）
  ├─ version（如v1.0, v1.1, v2.0）
  ├─ parent_version（指向上一版本，追溯演进）
  └─ standard_docs（引用的标准文档及版本）

AI生成物版本表（ai_artifacts）
  ├─ artifact_id（生成物唯一标识）
  ├─ domain_id（关联领域版本）
  ├─ artifact_type（聚类/矩阵/问卷等）
  └─ content（实际生成内容JSONB）

项目快照表（project_snapshots）
  ├─ project_id（项目唯一标识）
  ├─ domain_version（项目创建时使用的领域版本）
  └─ artifacts_snapshot（冻结的所有AI生成物）
```

**向下兼容策略：**

当IT标准更新（如ISO 27001:2013 → ISO 27001:2022）时：
1. 创建新版本领域（v2.0），保留旧版本（v1.0，标记deprecated）
2. 历史项目继续使用快照数据（只读），确保评估结果一致性
3. 提供"重新评估"功能 - 用户可选择将旧项目升级到新版本重新运行
4. 对标数据分版本存储，避免不同标准版本的数据混淆

## 风险应对与质量保障

基于事前风险分析（Pre-mortem Analysis），Csaas针对技术和质量风险建立了系统性应对机制：

### 技术可靠性保障

**异步执行与进度透明：**
- 三模型调用采用异步执行，咨询师提交任务后可继续其他工作，完成后邮件/短信通知
- 实时进度显示：显示"模型1已完成（3分12秒），模型2进行中（已用2分钟，预计还需3分钟），模型3排队中"
- 预期时间估算：基于历史数据预估"本次任务预计15-25分钟完成"

**多级降级策略：**
- **Level 1（理想）**：三模型全部成功 → 正常质量验证流程
- **Level 2（可接受）**：两个模型成功，一个失败 → 基于两个模型进行验证（降低一致性要求到70%）
- **Level 3（降级）**：仅一个模型成功 → 标记为"低置信度"，强制人工深度审核
- **Level 4（兜底）**：三个模型全失败 → 提供"手工模式"，AI辅助检索但不自动生成

**容错机制：**
- 自动重试：API超时或限流时，等待5-10分钟后自动重试（最多3次）
- 模型替换：某个模型持续失败时，自动切换到备用模型（如GPT-4失败时切换到GPT-4o-mini）
- 断点续传：长文档分段处理，部分失败时只重试失败的部分，不影响已成功的部分

### 质量保障体系

**质量保障的三重防线：**

**防线1：AI内部验证（三模型一致性）**
- 分层相似度验证（结构一致性≥90% + 语义等价性≥80% + 细节一致性≥60%）
- 基于独立验证原理和最小可验证集合理论

**防线2：高风险条款强制审核**
- **自动识别高风险条款**：
  - 关键词匹配：安全、合规、法律责任、数据保护、访问控制
  - 标准章节匹配：ISO 27001第5-8章（核心控制）、COBIT第3章（合规）
  - 客户行业匹配：金融、医疗、政府客户 → 所有合规条款强制审核

- **强制审核流程**（不可跳过）：
  - 即使三模型一致性100%，高风险条款仍需主咨询师**逐条确认**
  - 系统提供"挑战性问题"：如"该条款建议移除某控制措施，请确认：1) 标准确实废弃了该措施？2) 没有新的替代措施？3) 移除后不会引入新风险？"
  - 咨询师必须勾选"我已仔细核对原文，确认无误"才能继续

**防线3：外部验证**
- **权威解读库对照**：
  - 系统维护"标准权威解读库"（如ISO官方解读文档、ITSS实施指南）
  - AI生成结果后，自动对照权威解读，标记"与官方解读不一致"的条目

- **行业专家抽查**（成熟期）：
  - 每月随机抽取5-10个项目，邀请外部专家（独立咨询专家）审核
  - 专家发现错误 → 反馈到系统 → 优化模型提示词 → 建立"常见误判案例库"

- **客户反馈闭环**：
  - 跟踪报告实施效果：企业按照报告建议实施后，是否达到预期？
  - 如发现问题 → 回溯分析AI生成环节 → 优化算法

**法律责任界定：**
- **免责条款**：
  - "本系统为咨询辅助工具，最终报告质量由签字咨询师负责"
  - "系统提供的建议需结合企业实际情况判断，不构成法律或合规承诺"

- **责任分担机制**：
  - AI错误且咨询师未发现 → 平台承担30%责任（上限50万/项目）
  - AI正确但咨询师改错 → 咨询师承担100%责任
  - 建立"质量保证基金"，用于赔付潜在损失

**质量追踪与持续改进：**
- 建立"AI错误案例库"，记录所有被人工纠正的AI错误
- 每季度分析错误模式，优化提示词和验证算法
- 向用户透明披露"过去3个月AI准确率95.2%，比上季度提升1.3%"

### 风险矩阵与应对优先级

| 风险类型 | 发生概率 | 影响程度 | 应对优先级 | 核心策略 |
|---------|---------|---------|-----------|---------|
| 质量失败（客户索赔） | 中 | 极高 | **P0** | 三重防线验证 + 高风险条款强制审核 + 质量保证基金 |

**DEMO阶段重点：**
- ✅ 实现完整的三重防线验证机制

3. **成熟度矩阵生成** - 五分制矩阵（含细节溯源、关联分析、前置条件）+ 结构一致性验证
4. **问卷生成** - 多角色定制问卷 + 交叉验证设计 + 语义相似度验证
5. **落地措施生成** - 成熟度跃迁的具体工作清单

**质量保障机制：**
- 三模型并行调用架构（异步队列模式）
- 分层相似度验证（结构90%+语义80%+细节60%）
- 多级降级策略（Level 1-4）
- 覆盖率自动检查（列出遗漏条款）
- 高风险条款识别与强制审核

**人工审核流程：**
- 渐进式披露三阶段（独立思考 → 对比发现 → 深度验证）
- AI逻辑说明生成（为什么这样聚类）
- 原文对照显示（溯源到标准文档原文）
- 审核质量追踪（时间、修改率预警）

**基础数据管理：**
- 咨询领域版本管理（v1.0结构）
- 项目快照存储（保证历史数据一致性）
- 独占数据vs共享数据分层（数据主权设计）

**MVP不包含（留待后续）：**
- ❌ 行业对标功能（需要100+项目数据积累）
- ❌ 外部验证机制（权威解读库对照、专家抽查）
- ❌ 能力校准测试（混入故意错误）
- ❌ 多租户权限管理（DEMO阶段单租户即可）
- ❌ 问卷自动分发与进度追踪（手工操作）
- ❌ 报告自动生成与导出（手工整理）

### Growth Features - 成长功能（6-18个月）

**数据飞轮功能：**
- 行业对标数据可视化（100个项目后启用基础对标）

## 功能需求

### 功能需求概述

本节定义Csaas平台的核心能力，遵循"WHAT不是HOW"原则，描述系统必须提供的功能，而非具体实现方式。所有需求均以可测试的用户能力格式编写："[角色] 可以 [能力]"。

**需求组织结构：**
功能需求按照能力域(Capability Domain)组织，而非技术层(如前端/后端/数据库)。这种组织方式确保需求与用户价值直接对应，便于追溯用户旅程和业务目标。

**需求优先级标注：**
- **P0 (DEMO阶段必须)** - MVP验证核心假设所必需的能力
- **P1 (Growth阶段)** - 规模化运营和数据飞轮所需的能力
- **P2 (Expansion阶段)** - 市场扩展和生态建设所需的能力

## 非功能需求 (Non-Functional Requirements)

### NFR概述

非功能需求定义**系统质量属性**（HOW WELL），而非功能性（WHAT）。本节仅记录对Csaas产品成功至关重要的质量标准。

**选择性原则**: 只记录对本产品实际重要的NFR类别。未列出的类别不代表被忽视，而是优先级不足以纳入正式需求。

**优先级标注**:
- **P0 (DEMO阶段必须)** - MVP验证和市场准入所必需
- **P1 (Growth阶段)** - 规模化运营和商业可持续性所需
- **P2 (Expansion阶段)** - 市场扩展和竞争优势所需

**更新说明**: 本NFR基于专家团队（架构师、测试工程师、产品经理、UX设计师）的评审建议完善，特别加强了AI治理、数据质量、可观测性、UX可访问性等领域。

### 1. AI治理与质量要求

**NFR-AI1: AI共识一致性** [P0]
- **标准**: 三模型并行生成的结果达到共识的比例 ≥ 85%（分层相似度验证通过）
- **测量方式**: (通过相似度验证的任务数 / 总任务数) × 100%
- **业务影响**: 低于85%意味着AI质量不稳定，会增加人工审核负担
- **阶段目标**: DEMO阶段 ≥ 80%可接受，Growth阶段需达到 ≥ 85%

**NFR-AI2: 单模型响应时间SLA** [P0]
- **标准**: 单个AI模型处理标准文档（200-300页）的响应时间 P95 ≤ 15分钟
- **测量方式**: 95%分位的单模型API响应时间
- **业务影响**: 单模型慢会拖累整体任务完成时间，影响用户体验
- **降级策略**: 单模型超时15分钟后，继续等待其他2个模型（最多30分钟总超时）

**NFR-AI3: AI输出可解释性** [P1]
- **标准**: 所有AI生成结果必须包含可追溯的推理说明（为什么这样聚类、为什么给出某个评分）
- **测量方式**: 100%的AI生成结果包含逻辑说明，用户测试中90%的咨询师能理解AI推理
- **业务影响**: 不可解释的AI输出会降低咨询师信任度，增加审核难度

**NFR-AI4: AI模型漂移检测** [P1]
- **标准**: 系统自动检测AI模型质量退化（与baseline相比准确率下降 > 5%时告警）
- **测量方式**: 每周运行标准测试集（10个标杆项目），对比baseline准确率
- **业务影响**: 模型升级或API变更可能导致质量退化，需及时发现

**NFR-AI5: AI共识超时处理** [P0]
- **标准**: 当三模型分歧持续超过3次重试（总计30分钟）时，系统自动降级为"低置信度"模式，提示人工深度审核
- **测量方式**: 模拟分歧场景，验证降级逻辑正确触发
- **业务影响**: 无限等待共识会导致任务卡死

**NFR-AI6: 部分共识接受标准** [P0]
- **标准**: 当仅2个模型达成一致（相似度≥90%），第3个模型失败或分歧时，系统自动接受2模型共识并标记为"中等置信度"
- **测量方式**: 2-of-3共识场景的自动处理成功率 100%
- **业务影响**: 提高系统容错能力，避免单点故障导致任务失败

**NFR-AI7: AI成本异常检测** [P1]
- **标准**: 单个项目API成本超过正常值3倍时（正常≈10元，异常>30元）自动告警并暂停任务
- **测量方式**: 实时成本追踪，异常检测规则触发测试
- **业务影响**: 防止恶意攻击或配置错误导致成本失控

---

### 2. 性能要求

**NFR-P1: AI任务执行时间** [P0] ⬆️ *从P1提升*
- **标准**: AI生成的评估交付速度快于人工咨询师当前基线（当前基线: 4-8小时手工工作）
  - DEMO阶段: 三模型并行处理完成时间 P50 ≤ 25分钟，P95 ≤ 35分钟
  - Growth阶段: P50 ≤ 20分钟，P95 ≤ 30分钟
- **测量方式**: 从任务提交到三模型全部返回结果的时间统计
- **业务影响**: 慢速AI会毁掉demo转化率，潜在客户会将速度与竞品对比

**NFR-P2: 审核界面响应时间** [P0]
- **标准**: 用户交互操作（展开聚类详情、查看原文对照、切换模型结果）响应时间 P95 ≤ 2秒
- **测量方式**: 前端性能监控（如Web Vitals），95%分位响应时间
- **业务影响**: 审核流程需频繁切换视图，响应慢会显著降低审核效率

**NFR-P3: 对标数据查询性能** [P2] ⬇️ *从P1降级*
- **标准**: 行业对标数据查询（含百分位排名计算）响应时间 P95 ≤ 3秒（适用于1000+项目数据量）
- **测量方式**: 查询请求到结果返回的端到端时间
- **业务影响**: DEMO和Growth早期阶段数据量小，暂不需要此性能优化
- **触发条件**: Expansion阶段达到500+项目时启用此NFR

**NFR-P4: 并发任务处理能力** [P0]
- **标准**: 
  - DEMO阶段: 支持10个并发AI生成任务，响应时间 < 3秒
  - Growth阶段: 支持50个并发任务，响应时间退化 < 10%
  - Expansion阶段: 支持200个并发任务
- **测量方式**: 负载测试在目标并发下的响应时间和错误率
- **业务影响**: 高峰期多个咨询师同时使用时不能相互影响

**NFR-P5: 问卷填写加载性能** [P0]
- **标准**: 问卷页面Time to Interactive (TTI) ≤ 3秒（含50-100题的完整问卷）
- **测量方式**: Lighthouse性能评分，TTI指标测量
- **业务影响**: 被调研者在工作间隙填写，加载慢会导致放弃

**NFR-P6: 进度更新频率** [P0]
- **标准**: AI任务进度每30秒更新一次，显示当前完成百分比、预计剩余时间、当前执行步骤描述
- **测量方式**: 前端进度条更新频率和内容完整性检查
- **业务影响**: 20分钟AI任务无进度反馈会让用户以为系统卡死

### 3. 可靠性要求

**NFR-R1: 系统可用性** [P0]
- **标准**: 
  - DEMO阶段: 正常运行时间(uptime) ≥ 99.5%（每月计划外停机 ≤ 3.6小时）
  - Growth阶段: ≥ 99.9%（每月 ≤ 43分钟）
- **测量方式**: (总时间 - 计划外停机时间) / 总时间
- **业务影响**: 咨询项目有交付截止期，系统不可用会导致咨询师无法完成项目

**NFR-R2: AI任务完成率** [P0]
- **标准**: 三模型并行调用任务完成率（非降级模式）≥ 90%，通过端到端自动化测试每小时验证，4小时窗口内低于90%时告警
- **测量方式**: (成功完成的任务数 / 总任务数) × 100%，自动化监控
- **业务影响**: 降级模式会降低质量置信度，影响咨询师信任

**NFR-R3: 数据持久性** [P0] ⬆️ *从隐含P1提升*
- **标准**: 所有项目数据、AI生成内容、人工审核记录零丢失，数据完整性校验成功率 100%
- **测量方式**: 每日自动备份验证，季度备份恢复演练成功率 100%
- **业务影响**: 项目数据丢失会导致咨询公司客户索赔和声誉损失
- **备份策略**: 每日全量备份 + 每小时增量备份，保留90天

**NFR-R4: 降级模式可用性** [P0]
- **标准**: 降级模式必须可靠执行以下核心能力：
  - Level 2 (2模型成功): 相似度验证、人工审核界面、结果保存
  - Level 3 (1模型成功): 标记低置信度、强制人工深度审核、结果保存
  - Level 4 (全失败): 手工模式（AI辅助检索但不自动生成）
- **测量方式**: 混沌工程测试各级降级场景，功能可用性 100%
- **业务影响**: 降级机制失效会导致任务卡死，用户无法继续工作

**NFR-R5: 异步任务通知可靠性** [P0]
- **标准**: 任务完成后邮件/短信通知发送成功率 ≥ 95%，通过邮件服务商的delivery confirmation webhook验证
- **测量方式**: (成功发送确认的通知数 / 应发送总数) × 100%
- **业务影响**: 咨询师依赖通知来知道任务完成，通知失败会导致工作延误

**NFR-R6: 幂等性保证** [P1]
- **标准**: AI任务重试时保证幂等性（重复提交相同任务不会产生重复结果或重复扣费）
- **测量方式**: 模拟重试场景，验证结果唯一性和成本计算正确性
- **业务影响**: 非幂等重试会导致成本失控和数据混乱

**NFR-R7: 会话状态持久化** [P1]
- **标准**: 服务器重启时，进行中的问卷填写会话状态完全恢复（已填写内容、当前位置）
- **测量方式**: 模拟服务器重启，验证会话恢复成功率 100%
- **业务影响**: 会话丢失会导致被调研者重新填写，严重影响体验

### 4. 安全性要求

**NFR-S1: 数据传输加密** [P0]
- **标准**: 所有网络传输强制使用当前行业标准加密协议（最低安全等级: TLS 1.2或更高版本）
- **测量方式**: 安全扫描工具验证无明文HTTP端点，SSL Labs评分 ≥ A
- **业务影响**: 企业数据在传输中被截获会导致客户信任丧失
- **合规要求**: 符合等保2.0三级要求（Growth阶段）

**NFR-S2: 敏感数据存储加密** [P0]
- **标准**: 独占数据（企业名称、问卷回复、报告内容）使用行业标准加密算法（最低256位密钥长度）加密存储
- **测量方式**: 数据库审计确认敏感字段已加密，加密密钥独立管理
- **业务影响**: 数据库泄露时减少敏感信息暴露风险

**NFR-S3: 身份认证安全** [P0]
- **标准**: 
  - 密码强度: ≥8字符，包含大小写字母+数字+特殊符号
  - 登录失败锁定: 5次失败后锁定账号15分钟
  - 会话超时: 2小时无操作自动登出，登出前2分钟弹窗预警
- **测量方式**: 安全测试验证密码策略执行和会话管理
- **业务影响**: 弱密码或会话劫持会导致未授权访问客户数据

**NFR-S4: 多租户数据隔离** [P0]
- **标准**: 租户A的用户无法通过任何手段访问租户B的数据（4层防御机制，详见ADR #3），渗透测试跨租户数据访问成功率必须为0%
- **测量方式**: 季度第三方渗透测试，自动化安全扫描持续监控
- **业务影响**: 数据泄露会导致平台信任崩溃和法律责任
- **合规要求**: 符合PIPL和GDPR（如有国际客户）

**NFR-S5: API密钥安全存储** [P0]
- **标准**: 所有第三方API密钥安全存储且支持轮换，代码和配置文件中不得硬编码
- **测量方式**: 代码审查和密钥扫描工具验证，密钥轮换演练成功率 100%
- **业务影响**: API密钥泄露会导致成本失控和服务滥用

**NFR-S6: 审计日志完整性** [P0] ⬆️ *从P1提升*
- **标准**: 所有敏感操作记录审计日志（包含操作者、时间、操作类型、数据对象），日志保留1年，任何人（包括管理员）无法篡改或删除
- **测量方式**: 审计日志覆盖率检查，日志完整性验证（哈希校验）
- **业务影响**: 安全事件调查和合规审计需要完整的操作记录
- **覆盖范围**: 数据访问、权限变更、数据导出、数据删除、配置修改

### 5. 数据质量与验证要求

**NFR-DQ1: 数据验证完整性** [P0]
- **标准**: 所有用户输入数据（问卷响应、标准文档上传）必须通过格式验证和业务规则验证，畸形数据拒绝率 100%
- **测量方式**: 自动化测试注入畸形数据，验证系统拒绝并提供明确错误提示
- **业务影响**: 畸形数据进入系统会导致AI生成失败或产生错误结果

**NFR-DQ2: 跨租户数据一致性** [P1]
- **标准**: 共享数据（去标识化对标数据）在所有租户视图中保持一致性，数据更新后5分钟内全局可见
- **测量方式**: 数据一致性检查脚本，多租户数据同步延迟测量
- **业务影响**: 数据不一致会导致对标结果错误

**NFR-DQ3: 数据迁移零损坏** [P1]
- **标准**: 数据库schema变更时，数据迁移前后数据完整性校验通过率 100%，零数据损坏
- **测量方式**: 迁移前后数据checksum对比，关键字段值验证
- **业务影响**: 数据损坏会导致历史项目无法访问

**NFR-DQ4: 问卷响应完整性验证** [P0]
- **标准**: 问卷提交时验证必填题完整性，部分填写的问卷自动保存为草稿（不计入统计）
- **测量方式**: 提交验证规则测试，草稿数据隔离验证
- **业务影响**: 不完整问卷计入统计会导致评估结果错误

### 6. 可观测性与监控要求

**NFR-OBS1: 错误诊断能力 (MTTD)** [P1]
- **标准**: 生产环境错误发生时，开发团队可在10分钟内定位根因（通过日志、监控、分布式追踪）
- **测量方式**: 平均故障定位时间(MTTD)统计，所有错误包含correlation ID、stack trace、业务上下文
- **业务影响**: 快速诊断可以减少系统停机时间和用户影响

**NFR-OBS2: 故障恢复能力 (MTTR)** [P1]
- **标准**: 平均故障恢复时间(MTTR) ≤ 30分钟（从告警触发到服务恢复）
- **测量方式**: 故障响应时间统计，分级处理（P0故障15分钟，P1故障30分钟，P2故障2小时）
- **业务影响**: 快速恢复减少业务中断影响

**NFR-OBS3: 分布式日志聚合** [P1]
- **标准**: AI任务的完整执行链路（任务提交→三模型调用→相似度计算→结果保存）可通过correlation ID追踪，日志聚合延迟 ≤ 30秒
- **测量方式**: 日志查询工具验证完整链路可追溯性
- **业务影响**: 分布式任务调试困难，需要完整链路日志

**NFR-OBS4: 系统健康仪表板** [P1]
- **标准**: 实时监控仪表板显示关键指标：AI任务成功率、API响应时间、系统可用性、成本消耗，数据刷新延迟 ≤ 1分钟
- **测量方式**: 仪表板功能测试，数据准确性验证
- **业务影响**: 运维团队需要实时了解系统健康状态

**NFR-OBS5: AI模型健康检查** [P0]
- **标准**: 每5分钟对所有AI模型API执行健康检查（简单请求测试），连续3次失败时告警并触发降级
- **测量方式**: 健康检查机制测试，告警触发验证
- **业务影响**: AI API是核心依赖，故障需要及时发现

### 7. 业务连续性与灾难恢复

**NFR-BC1: 灾难恢复目标** [P1]
- **标准**: 
  - RTO (恢复时间目标): ≤ 4小时（从灾难发生到服务恢复）
  - RPO (恢复点目标): ≤ 1小时（最大数据丢失时间窗口）
- **测量方式**: 季度灾难恢复演练，验证RTO/RPO达标
- **业务影响**: 重大灾难（如数据中心故障）时的业务连续性保障

**NFR-BC2: 备份恢复测试频率** [P0]
- **标准**: 每季度执行一次完整备份恢复演练，验证备份数据完整性和恢复流程有效性
- **测量方式**: 演练记录和成功率统计
- **业务影响**: 未经验证的备份可能在需要时无法恢复

**NFR-BC3: 分级事件响应时间** [P1]
- **标准**: 
  - P0事件（系统不可用）：15分钟内响应，1小时内初步方案
  - P1事件（核心功能受损）：30分钟内响应，4小时内初步方案
  - P2事件（次要功能问题）：2小时内响应，1个工作日内方案
- **测量方式**: 事件响应时间统计和SLA达标率
- **业务影响**: 快速响应减少客户影响和声誉损失

### 8. 可扩展性要求

**NFR-SC1: 用户规模支撑** [P0]
- **标准**: 
  - DEMO阶段: 1-2家咨询公司，5-10个并发用户，响应时间 < 3秒
  - Growth阶段: 10-15家咨询公司，50个并发用户，响应时间退化 < 10%
  - Expansion阶段: 50+家咨询公司，200个并发用户
- **测量方式**: 负载测试验证目标并发下的响应时间和错误率
- **业务影响**: 用户增长超出系统容量会导致性能下降和用户流失

**NFR-SC2: 数据增长处理** [P2] ⬇️ *从P1降级*
- **标准**: 系统在数据量增长10倍时（从100个项目到1000个项目），关键查询性能(P95)降低 < 10%
- **测量方式**: 基准测试在不同数据规模下的查询性能
- **业务影响**: Expansion阶段前无需关注此问题，有实际规模问题时再优化
- **触发条件**: 达到500个项目时启用此NFR

**NFR-SC3: AI API成本可控性** [P1] ⚠️ *保持P1不变（用户要求）*
- **标准**: 单个项目平均API调用成本 ≤ 10元人民币（三模型×5环节≈22次调用）
- **测量方式**: 每月API账单 / 完成项目数
- **业务影响**: 成本失控会侵蚀利润空间，影响商业模式可持续性
- **成本优化**: token限制、结果缓存、模型选择优化

**NFR-SC4: 水平扩展能力** [P1]
- **标准**: 系统核心服务（API服务器、Worker队列）采用无状态设计，支持通过增加实例数提升容量
- **测量方式**: 架构审查确认无状态设计，负载均衡测试
- **业务影响**: Growth阶段需要快速响应用户增长

**NFR-SC5: 缓存命中率优化** [P2]
- **标准**: 重复或相似标准文档的AI生成结果缓存命中率 ≥ 30%（Expansion阶段）
- **测量方式**: 缓存命中率监控统计
- **业务影响**: 缓存可降低API成本和响应时间

### 9. 可用性与用户体验要求

**NFR-U1: 新用户学习曲线** [P0] ⬆️ *从隐含P1提升*
- **标准**: 新咨询师首次使用渐进式披露审核界面时，80%的测试用户无需培训即可在30分钟内完成一个环节的审核（n≥10用户per测试）
- **测量方式**: 用户测试观察新用户首次使用的完成时间、求助次数、任务成功率
- **业务影响**: 复杂界面会降低咨询师采用意愿，第一印象决定成交率

**NFR-U2: 错误信息可理解性** [P0]
- **标准**: 所有错误提示使用业务语言（非技术术语）并提供明确的解决建议，用户测试中90%的用户能理解错误信息并采取正确行动（无需联系支持）
- **示例**: 
  - ❌ "API Gateway 504 timeout"
  - ✅ "AI模型处理超时，请稍后重试或联系支持团队（错误代码: T504）"
- **测量方式**: 用户测试验证，支持工单数量跟踪
- **业务影响**: 技术错误信息会让非技术用户困惑和沮丧

**NFR-U3: 移动端问卷填写体验** [P2] ⬇️ *从隐含P1降级*
- **标准**: 问卷填写界面在移动设备上完全可用（iOS 14+, Android 10+），System Usability Scale评分 ≥ 70
- **测量方式**: 响应式设计测试（320px - 768px宽度），SUS问卷评分
- **业务影响**: B2B IT咨询评估主要在桌面完成，移动端优先级较低

**NFR-U4: 进度透明度** [P0]
- **标准**: 用户在所有长时间操作（AI生成、问卷收集、报告生成）中始终清楚"当前在哪个环节"、"已完成X%"、"预计还需Y分钟"
- **测量方式**: 界面设计审查确认进度指示器存在、准确且每30秒更新
- **业务影响**: 不清楚进度会让用户焦虑，可能放弃操作

**NFR-U5: 会话超时预警** [P0]
- **标准**: 会话即将超时（剩余2分钟）时弹窗预警，用户可选择"延长会话"或"保存并退出"
- **测量方式**: 会话管理功能测试
- **业务影响**: 突然登出会导致用户丢失未保存工作，引发沮丧

**NFR-U6: 自动保存频率** [P0]
- **标准**: 问卷填写和审核编辑内容每60秒自动保存草稿，用户手动操作（点击下一题）时立即保存
- **测量方式**: 自动保存机制测试，模拟浏览器崩溃后恢复验证
- **业务影响**: 无自动保存会导致用户工作丢失

**NFR-U7: 键盘导航完整性** [P0] 🆕
- **标准**: 审核界面的所有核心操作支持键盘快捷键（无需鼠标即可完成审核工作流）
- **测量方式**: 键盘导航测试覆盖所有核心流程
- **业务影响**: 企业客户（特别是政府、金融）要求无障碍访问基础支持

**NFR-U8: 屏幕阅读器兼容性** [P0] 🆕
- **标准**: 核心用户旅程（项目创建、AI审核、问卷填写）与主流屏幕阅读器兼容（NVDA、JAWS）
- **测量方式**: 屏幕阅读器测试验证关键流程可用性
- **业务影响**: 企业采购时会检查基础无障碍支持

### 10. 集成与互操作性要求

**NFR-I1: AI API容错与降级** [P0]
- **标准**: 
  - 单个模型API故障时，自动切换备用API密钥（切换时间 ≤ 10秒）
  - 备用密钥失败时，启动降级模式继续运行
- **测量方式**: 混沌工程测试模拟API故障，验证自动切换和降级
- **业务影响**: AI模型API是核心依赖，故障会直接导致系统不可用

**NFR-I2: 邮件服务可替换性** [P1]
- **标准**: 系统支持快速切换邮件服务提供商（SendGrid、阿里云、AWS SES），配置变更和验证 ≤ 5分钟停机时间
- **测量方式**: 邮件服务抽象层架构审查，服务切换演练
- **业务影响**: 邮件服务故障或费用调整时需要快速迁移

**NFR-I3: 第三方服务降级** [P0]
- **标准**: 当非核心第三方服务故障时，系统核心功能不受影响（核心功能明确定义为：AI生成、人工审核、数据保存）
- **测量方式**: 模拟邮件、短信服务故障，验证核心功能可用性
- **业务影响**: 避免第三方服务故障导致整个系统不可用

**NFR-I4: API版本兼容性** [P1]
- **标准**: 外部API（OpenAI、Anthropic）版本升级时，系统保持向后兼容30天，通过自动化contract tests验证兼容性
- **测量方式**: API版本锁定机制和渐进式升级流程，contract tests覆盖
- **业务影响**: API突然升级导致功能中断会影响正在进行的咨询项目

**NFR-I5: 网络分区处理** [P1] 🆕
- **标准**: 当国产模型与海外模型（GPT-4/Claude）网络隔离时，系统自动检测并启用单一模型组降级模式
- **测量方式**: 模拟网络分区场景，验证系统自适应降级
- **业务影响**: 中国网络环境下可能出现海外API访问受限

### 11. 可维护性要求

**NFR-M1: 配置热更新** [P1]
- **标准**: 关键配置（API密钥、相似度阈值、降级策略参数）支持热更新，配置变更生效时间 ≤ 30秒，无需重启服务
- **测量方式**: 配置管理系统支持测试，配置变更生效时间测量
- **业务影响**: 紧急配置调整（如API密钥泄露更换）需要立即生效

**NFR-M2: 代码可测试性** [P1] ⬇️ *从隐含P0降级*
- **标准**: 
  - DEMO阶段: 核心业务逻辑（AI任务编排、相似度计算、权限控制）代码测试覆盖率 ≥ 60%
  - Growth阶段: 整体代码测试覆盖率 ≥ 80%
- **测量方式**: 代码覆盖率工具（如Jest、pytest-cov）
- **业务影响**: 高测试覆盖率降低回归风险，支持快速迭代

**NFR-M3: 蓝绿部署能力** [P1] 🆕
- **标准**: 系统支持蓝绿部署或金丝雀发布，新版本部署过程中零停机时间
- **测量方式**: 部署流程测试，用户会话不中断验证
- **业务影响**: Growth阶段需要频繁发布新功能，零停机部署提升用户体验

**NFR-M4: Feature Flag基础设施** [P1] 🆕
- **标准**: 新功能通过Feature Flag控制，可随时禁用有问题的功能（kill switch），flag切换生效时间 ≤ 1分钟
- **测量方式**: Feature flag系统功能测试
- **业务影响**: 发布后发现问题时可快速回退功能，无需完整回滚

### 12. 合规性要求

**NFR-C1: 数据保护合规** [P0] ⬆️ *从P1提升*
- **标准**: 
  - DEMO阶段: 符合《个人信息保护法》(PIPL)基本要求
  - Growth阶段: 符合等保2.0三级认证要求
  - 如有国际客户: 符合GDPR基本要求（数据最小化、知情同意、被遗忘权）
- **测量方式**: 合规审计和认证流程
- **业务影响**: 不合规会导致法律风险，PIPL合规是中国市场准入必需

**NFR-C2: 数据留存与删除** [P0] ⬆️ *从P1提升*
- **标准**: 
  - 咨询公司可随时导出所有自有数据（完整数据包 ≤ 24小时交付）
  - 咨询公司注销后，独占数据在30天内完全删除（符合PIPL"被遗忘权"），删除操作可审计
  - 共享数据（已去标识化）保留用于对标分析
- **测量方式**: 数据删除流程测试和审计日志验证，数据导出完整性验证
- **业务影响**: PIPL要求的强制性条款，不符合无法在中国运营

**NFR-C3: 审计追溯能力** [P0]
- **标准**: 所有数据访问、导出、删除操作可追溯到具体用户和时间（审计日志保留1年），支持监管机构审计查询
- **测量方式**: 审计日志完整性检查，查询功能测试
- **业务影响**: 合规审计和安全事件调查需要完整的操作记录

**NFR-C4: 数据本地化** [P1] 🆕
- **标准**: 中国客户的所有数据（包括AI生成任务）存储在中国境内数据中心，不出境
- **测量方式**: 数据流向审计，基础设施架构审查
- **业务影响**: 部分监管行业（金融、政府）要求数据本地化

### NFR验证策略

**性能验证：**
- 负载测试工具（k6、JMeter）模拟目标并发
- APM工具（Datadog、New Relic）持续监控生产性能
- 每月性能基准测试，追踪性能演进趋势

**可靠性验证：**
- 混沌工程测试（故意触发故障，验证降级机制）
- 季度备份恢复演练和灾难恢复演练
- 正常运行时间监控（UptimeRobot、Pingdom）

**安全性验证：**
- Growth阶段前进行第三方渗透测试
- 自动化安全扫描（SAST/DAST工具集成CI/CD）
- Expansion阶段第三方安全审计

**AI质量验证：**
- 每周自动化回归测试（10个标杆项目）
- AI模型A/B测试（新模型vs旧模型质量对比）
- 定期人工抽查（DEMO 100%，Growth 30%，Expansion 10%）

**可用性验证：**
- 每季度用户测试（n≥10新用户）
- 无障碍测试（键盘导航、屏幕阅读器）
- System Usability Scale (SUS) 问卷评分
