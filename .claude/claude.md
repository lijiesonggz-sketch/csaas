# 项目自定义指令

本文件包含针对此项目的特定规则和指南，用于改善 Claude Code 的工作流程。

---

## 核心交互原则（基于对话历史分析）

### 语言设置

- ✅ **始终使用中文回复**，即使用户用英文提问
- ✅ 技术术语保留英文（API、PRD、MVP、EPIC、Story 等）
- ❌ **绝对不要**用英文回复（除非用户明确要求）

### 执行方式

- ✅ **全自动执行**：收到指令后直接执行，不要在每个步骤后停顿询问确认
- ✅ **批量完成后统一报告**：完成多个步骤后再报告进度和结果
- ✅ **遇到阻塞问题才询问**：只有在真正需要用户决策时才停下来
- ❌ **绝对不要**因 Token/时间限制而降低任务完成质量，不要简化实现
- ❌ **绝对不要**在故事之间停顿等待确认
- ❌ **绝对不要**反复询问已经明确的事项

**用户典型指令示例**：

- "直接执行，不用再和我交互"
- "记住全自动"
- "你做完所有，我再手工验收测试"

### 代码复用原则

- ✅ **优先复用已有实现**：修改前主动查看相关代码
- ✅ **明确参考之前的实现**：当用户说"参考 X 文件"时，直接复用而非重写
- ✅ **如需重写，说明理由**：在无法复用的情况下解释原因
- ❌ **绝对不要**默认重写代码
- ❌ **绝对不要**绕开已有成功的代码重新写逻辑

### 进度反馈要求

- ✅ **长时间任务（>30秒）主动报告进度**：使用进度条或步骤指示器
- ✅ **每完成一个主要步骤后报告**：告知完成了什么，下一步做什么
- ✅ **预计耗时任务提前告知**：在开始前说明预计时间和步骤数量
- ❌ **绝对不要**静默执行让用户等待而不反馈

**用户痛点**："为什么停住了"、"为什么一直执行不下去，20多分钟了"

### 测试与质量标准

- ✅ **严格遵循 TDD**：测试覆盖率 > 80%，先写测试再写代码
- ✅ **质量优先于时间和Token消耗**：不因限制而简化实施
- ✅ **不跳过任何测试步骤**：包括单元测试、集成测试、代码审查
- ✅ **深入分析错误**：查看后台日志，分析根本原因，不表面修复
- ❌ **绝对不要**因 Token/时间限制而跳过测试
- ❌ **绝对不要**为了完成而降低标准

**用户典型纠正**：

- "你2.1是不需要测试的吗？为啥跳过automatetest"
- "不要因Token限制而简化"
- "程序质量优于TOKEN和时间消耗"

### 技术栈规范

| 领域     | 技术选择                                         |
| ------ | -------------------------------------------- |
| 前端     | Next.js + React + TypeScript + **shadcn/ui** |
| 后端     | NestJS + PostgreSQL + TypeORM                |

- ✅ **优先使用 shadcn/ui**：项目正在从 MUI 迁移到 shadcn/ui
- ✅ **使用 TypeScript**：严格的类型检查

### 上下文管理

- ✅ **保持上下文清洁**：长对话后主动总结关键决策
- ✅ **重要实现细节持久化**：记录在文档中避免丢失
- ✅ **阶段性 clear**：完成一个阶段后使用 `/clear` 或新建 subagent
- ❌ **绝对不要**忘记之前的讨论或实现

**用户痛点**："你不是已经12月30前就写好了所有的独立页面和功能了吗？为什么又从头开始写逻辑"

## 测试指南

### 修复测试的正确方法

**关键原则：测试必须适应代码，而不是代码适应测试**

- ❌ **绝对不要**修改生产代码来让测试通过
- ❌ **绝对不要**在生产代码中添加 `data-testid` 或其他仅用于测试的属性
- ✅ **必须**通过查看实际的生产代码来找到正确的选择器
- ✅ **必须**修改测试代码以匹配实际的 UI/代码实现
- ✅Run E2E tests after completing each story implementation
- ✅Check test data passwords and database schema before running tests
- ✅Use Playwright for visual debugging when layout issues occur
- ✅Verify all tests pass before marking story as done

**示例：**

```typescript
// ❌ 错误做法：修改生产代码
<button data-testid="submit-button">提交</button>

// ✅ 正确做法：修改测试以匹配实际代码
await page.getByRole('button', { name: '提交' }).click()
```

### 测试失败调试流程

当遇到测试失败时，必须明确区分问题类型：

1. **测试代码问题**（优先处理）
   
   - 错误的选择器
   - 过时的断言
   - 测试数据不匹配

2. **生产代码 bug**（需要用户确认）
   
   - 实际功能损坏
   - API 返回错误
   - 业务逻辑问题

**调试前必须询问：**
"这个测试失败是因为测试代码需要更新，还是因为生产代码有 bug？"

### 测试文件安全规则

- ❌ **绝对不要**删除测试文件
- ❌ **绝对不要**跳过测试步骤（automate-test、code-review）
- ❌ **绝对不要**因时间/Token限制而跳过测试
- ✅ 如果测试文件损坏无法修复，使用 `git revert` 或 `git restore` 恢复
- ✅ 如果不确定如何修复，保留文件并标记为 `test.skip()`

### 测试覆盖率要求

- ✅ **测试覆盖率 > 80%**
- ✅ **TDD 方式开发**：先写测试用例，再写代码
- ✅ **包含所有测试类型**：单元测试、集成测试、E2E 测试
- ✅ **不因 Token/时间压力降低标准**
- ❌ **绝对不要**为了测试通过而降低测试标准

## 通用行为规则

### 停止命令响应

当用户说"停止"、"stop"、"halt"或要求停止时：

- ✅ **立即停止**当前操作
- ❌ **不要**继续"再试一次"或"最后一步"
- ❌ **不要**完成"当前迭代"
- 简单回复："已停止。需要我做什么？"

### 长时间运行的工作流

对于可能运行超过 30 分钟的任务：

- ✅ **在开始前告知预计的步骤数量和预计时间**
- ✅ **每完成一个主要步骤后主动报告进度**（不要等用户询问）
- ✅ **使用进度指示器**：如 "步骤 3/10 完成"
- ✅ **提供明确的检查点**，允许用户中途验证
- ❌ **绝对不要**静默执行超过 2 分钟不反馈
- ❌ **绝对不要**在没有检查点的情况下运行超过 20 个工具调用

**用户痛点提醒**：

- "为什么停在这里10分钟了"
- "为什么一直执行不下去，20多分钟了"
- "你在做什么？为什么执行了40分钟还不完？"

## 文件操作规则

### 大文件写入

当文件内容超过 50KB 或 1000 行时：

- ❌ **不要**使用 Write 工具（会失败）

- ✅ **使用** Bash 命令进行分段写入：
  
  ```bash
  cat > file.txt << 'EOF'
  [内容第一部分]
  EOF
  
  cat >> file.txt << 'EOF'
  [内容第二部分]
  EOF
  ```

### 文件编辑限制

- 单次 Write 操作不要超过 50 行
- 如果需要写入更多内容，先写入前 50 行，然后使用 Edit 工具添加剩余内容

## 安全指南

### 破坏性更改前的检查

在对任何文件进行重大修改之前：

1. **检查 git 状态**
   
   ```bash
   git status [文件路径]
   ```

2. **如果文件未被 git 跟踪**
   
   - 创建备份：`cp file.ts file.ts.backup`
   - 或者先提交：`git add file.ts && git commit -m "backup before changes"`

3. **绝对不要**
   
   - 删除未被 git 跟踪的文件
   - 在没有备份的情况下进行破坏性编辑

### 代码审查循环限制

- 最多进行 **2 轮**代码审查迭代
- ✅ **自动修复所有 HIGH 和 MEDIUM 级别问题**
- ✅ **不希望在审查阶段等待人工确认**
- 审查完成后直接实施修复

### 错误处理深度要求

- ✅ **查看后台日志**：遇到错误时先查看详细日志
- ✅ **分析控制台错误**：提供具体的错误信息
- ✅ **根因分析**：深入分析根本原因，不表面修复
- ✅ **避免重复错误**：修复后确保同一问题不会再次出现
- ❌ **绝对不要**表面修复而不深入分析

**用户典型纠正**：

- "还是无法生成，你已经改了几次了，你重新仔细分析看看为什么"
- "还是报错，没生成" / "仍然失败"

## E2E 测试特定规则（Playwright）

### 浏览器自动化工具选择

当需要进行网页测试、截图或浏览器自动化时：

- **必须优先使用** `playwright-cli` skill
  - 使用 `playwright-cli open` 打开浏览器
  - 使用 `playwright-cli click` 点击元素
  - 使用 `playwright-cli type` 输入文本
  - 使用 `playwright-cli screenshot` 截图
  - 其他 playwright-cli 命令
- **不要使用** MCP 的 playwright 工具（如 `mcp__playwright__browser_navigate` 等）
- 例外情况：当用户明确要求使用 MCP 工具时

### 测试数据准备

- 在运行测试前，确认测试数据已正确设置
- 检查数据库中是否存在必要的测试用户和数据
- 如果测试失败提示"找不到元素"，首先检查是否是登录问题

### 选择器策略

优先级顺序：

1. `getByRole()` - 最稳定
2. `getByLabel()` - 表单元素
3. `getByText()` - 文本内容
4. `getByTestId()` - 最后选择（需要修改生产代码）

### 测试失败分类

- **登录/认证问题** → 检查测试用户凭据和会话状态
- **选择器问题** → 读取实际页面代码，更新选择器
- **时序问题** → 添加适当的 `waitFor` 调用
- **数据问题** → 检查测试数据设置脚本

当用户说以下话时，表示满意：

- "聊的很爽，给了我很多维度的思考"

- 用中文回复，技术术语保留英文

## BMAD 工作流程规范

### EPIC/Story 执行流程

当使用 BMAD 框架执行 EPIC 时，遵循以下流程：

```
create → validate → dev → automate-test → code-review → DONE
```

- ✅ **全自动执行**：Story 之间不停顿等待确认
- ✅ **完整的五个流程**：确保测试自动化和代码审查都执行
- ✅ **文档同步**：代码变更时同步更新相关文档
- ❌ **绝对不要**在五个流程未完成时直接改状态为 DONE

**用户典型纠正**：

- "为什么你story和story之间会停在那里等我"
- "story2.1里面的（创建→验证→开发→测试自动化→代码审查）这五个流程出现了后两个流程没做就直接改状态为DONE"

## 质量完成原则

### 高质量完成要求

1. **不要受 TOKEN 限制而快速完成任务**
   
   - 每个文件都需要仔细、完整地重构
   - 不要使用批处理（sed、grep 等）跳过详细检查
   - 逐个文件处理，确保每个细节都正确
   - 完成后验证代码质量

2. **渐进式完成**
   
   - 完成一个任务就更新 REFACTORING-TODO.md 文件
   - 标记完成的任务清单
   - 防止因 Token 重置而丢失进度

3. **质量验证**
   
   - 每个文件重构完成后，检查：
     - 所有 MUI 组件导入正确
     - 所有 className 替换为 sx 属性
     - 图标迁移完整
     - 没有遗漏的旧组件引用
   - 设计系统一致（颜色、圆角、阴影）

4. **不要跳过任何文件**
   
   - 即使文件很大，也要仔细完整处理
   - 不要因为时间压力而降低质量

### 文档同步要求

- ✅ **代码变更必须同步更新文档**
- ✅ **避免文档与代码不一致**
- ✅ **记录技术债**
- ❌ **绝对不要**遗漏文档更新
