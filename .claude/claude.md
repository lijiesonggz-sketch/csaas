# 项目自定义指令

本文件包含针对此项目的特定规则和指南，用于改善 Claude Code 的工作流程。

## 测试指南

### 修复测试的正确方法

**关键原则：测试必须适应代码，而不是代码适应测试**

- ❌ **绝对不要**修改生产代码来让测试通过
- ❌ **绝对不要**在生产代码中添加 `data-testid` 或其他仅用于测试的属性
- ✅ **必须**通过查看实际的生产代码来找到正确的选择器
- ✅ **必须**修改测试代码以匹配实际的 UI/代码实现

**示例：**

```typescript
// ❌ 错误做法：修改生产代码
<button data-testid="submit-button">提交</button>

// ✅ 正确做法：修改测试以匹配实际代码
await page.getByRole('button', { name: '提交' }).click()
```

### 测试失败调试流程

当遇到测试失败时，必须明确区分问题类型：

1. **测试代码问题**（优先处理）
   
   - 错误的选择器
   - 过时的断言
   - 测试数据不匹配

2. **生产代码 bug**（需要用户确认）
   
   - 实际功能损坏
   - API 返回错误
   - 业务逻辑问题

**调试前必须询问：**
"这个测试失败是因为测试代码需要更新，还是因为生产代码有 bug？"

### 测试文件安全规则

- ❌ **绝对不要**删除测试文件
- ✅ 如果测试文件损坏无法修复，使用 `git revert` 或 `git restore` 恢复
- ✅ 如果不确定如何修复，保留文件并标记为 `test.skip()`

## 通用行为规则

### 停止命令响应

当用户说"停止"、"stop"、"halt"或要求停止时：

- ✅ **立即停止**当前操作
- ❌ **不要**继续"再试一次"或"最后一步"
- ❌ **不要**完成"当前迭代"
- 简单回复："已停止。需要我做什么？"

### 长时间运行的工作流

对于可能运行超过 30 分钟的任务：

- ✅ 在开始前告知预计的步骤数量
- ✅ 每完成一个主要步骤后报告进度
- ✅ 提供明确的检查点，允许用户中途验证
- ❌ 不要在没有检查点的情况下运行超过 20 个工具调用

## 文件操作规则

### 大文件写入

当文件内容超过 50KB 或 1000 行时：

- ❌ **不要**使用 Write 工具（会失败）

- ✅ **使用** Bash 命令进行分段写入：
  
  ```bash
  cat > file.txt << 'EOF'
  [内容第一部分]
  EOF
  
  cat >> file.txt << 'EOF'
  [内容第二部分]
  EOF
  ```

### 文件编辑限制

- 单次 Write 操作不要超过 50 行
- 如果需要写入更多内容，先写入前 50 行，然后使用 Edit 工具添加剩余内容

## 安全指南

### 破坏性更改前的检查

在对任何文件进行重大修改之前：

1. **检查 git 状态**
   
   ```bash
   git status [文件路径]
   ```

2. **如果文件未被 git 跟踪**
   
   - 创建备份：`cp file.ts file.ts.backup`
   - 或者先提交：`git add file.ts && git commit -m "backup before changes"`

3. **绝对不要**
   
   - 删除未被 git 跟踪的文件
   - 在没有备份的情况下进行破坏性编辑

### 代码审查循环限制

- 最多进行 **2 轮**代码审查迭代

## E2E 测试特定规则（Playwright）

### 测试数据准备

- 在运行测试前，确认测试数据已正确设置
- 检查数据库中是否存在必要的测试用户和数据
- 如果测试失败提示"找不到元素"，首先检查是否是登录问题

### 选择器策略

优先级顺序：

1. `getByRole()` - 最稳定
2. `getByLabel()` - 表单元素
3. `getByText()` - 文本内容
4. `getByTestId()` - 最后选择（需要修改生产代码）

### 测试失败分类

- **登录/认证问题** → 检查测试用户凭据和会话状态
- **选择器问题** → 读取实际页面代码，更新选择器
- **时序问题** → 添加适当的 `waitFor` 调用
- **数据问题** → 检查测试数据设置脚本
